# 电商用户行为分析项目：基于 MySQL 的用户行为与转化漏斗分析

本项目基于电商平台用户行为数据，使用 MySQL 对用户浏览、收藏、加购、购买等行为进行分析，围绕数据清洗、整体行为表现、转化漏斗、用户分层、商品/类目表现和复购情况展开，模拟数据分析岗位中常见的业务分析流程。

## 1. 项目背景

在电商平台运营中，用户从浏览商品到最终购买通常会经历多个行为环节，例如浏览、收藏、加购和购买。不同环节之间的转化情况，可以反映平台流量质量、商品吸引力、用户购买意愿以及运营策略效果。

因此，本项目以电商用户行为数据为基础，分析用户在平台中的行为特征和转化路径，尝试发现用户活跃规律、关键流失环节、热门商品/类目以及具有运营价值的用户群体，为后续精细化运营提供数据参考。

## 2. 项目目标

本项目主要围绕以下问题展开：

1. 原始数据是否存在缺失值、异常行为类型和重复记录？
2. 平台整体用户行为结构如何？浏览、收藏、加购、购买分别占比多少？
3. 平台整体 PV、UV、购买次数、购买用户数和购买转化率表现如何？
4. 用户从浏览到收藏、加购、购买的转化漏斗表现如何？
5. 用户在不同日期和小时段的活跃规律如何？
6. 哪些用户属于高活跃用户、高价值用户、潜力用户和低意向浏览用户？
7. 哪些商品和类目更受欢迎、销量更高或转化率更好？
8. 平台购买用户的复购情况如何？

## 3. 数据说明

本项目使用的用户行为数据包含以下字段：

| 字段名 | 含义 |
|---|---|
| `user_id` | 用户 ID |
| `item_id` | 商品 ID |
| `category_id` | 商品类目 ID |
| `behavior_type` | 用户行为类型 |
| `time_stamp` | 用户行为发生时间戳 |

其中，`behavior_type` 包含四类行为：

| 行为类型 | 含义 |
|---|---|
| `pv` | 浏览 |
| `fav` | 收藏 |
| `cart` | 加购 |
| `buy` | 购买 |

本项目将原始时间戳转换为标准时间字段 `event_time`，并将分析周期限定为：

```text
2017-11-25 00:00:00 至 2017-12-03 23:59:59
```

## 6. 分析内容

### 6.1 数据质量检查

主要检查内容包括：

- 原始数据总量
- 关键字段缺失情况
- 行为类型是否异常
- 时间戳范围是否合理
- 是否存在完全重复记录

对应 SQL 文件：`sql/01_data_quality_check.sql`

### 6.2 数据清洗

清洗规则包括：

1. 过滤关键字段为空的记录。
2. 仅保留 `pv`、`fav`、`cart`、`buy` 四类有效行为。
3. 将分析周期限定在 2017-11-25 至 2017-12-03。
4. 使用 `SELECT DISTINCT` 去除完全重复记录。
5. 生成清洗后的分析表 `userbehavior_clean`。
6. 为用户、时间、商品、类目、行为类型等高频查询字段添加索引。

对应 SQL 文件：`sql/02_data_cleaning.sql`

### 6.3 整体行为分析

分析指标包括：

- 各行为类型数量
- 各行为类型占比
- PV
- UV
- 购买次数
- 购买用户数
- 用户购买转化率
- 每日行为趋势

对应 SQL 文件：`sql/03_overall_behavior_analysis.sql`

### 6.4 转化漏斗分析

本项目从两个口径分析转化漏斗：

1. **行为次数漏斗**  
   分析浏览、收藏、加购、购买等行为次数之间的转化关系。

2. **用户人数漏斗**  
   分析有浏览、收藏、加购、购买行为的去重用户数之间的转化关系。

核心指标包括：

- 收藏 / 浏览转化率
- 加购 / 浏览转化率
- 购买 / 浏览转化率
- 购买 / 加购转化率

对应 SQL 文件：`sql/04_conversion_funnel_analysis.sql`

### 6.5 用户分析

用户分析主要包括：

- 用户活跃度 TOP 20
- 高活跃用户行为结构拆分
- 不同小时段用户行为分布
- 各行为类型高峰时段
- 整体用户行为高峰期
- 用户行为汇总视图 `user_behavior_summary`

基于用户行为汇总结果，将用户初步划分为：

| 用户类型 | 划分规则 | 业务含义 |
|---|---|---|
| 高价值用户 | 购买次数 >= 10 | 购买频次高，适合会员维护和复购激励 |
| 潜力用户 | 有收藏或加购，但未购买 | 有购买兴趣，适合优惠券、降价提醒、购物车召回 |
| 浏览用户 | 只有浏览行为 | 购买意向较弱，适合优化推荐和商品展示 |

对应 SQL 文件：`sql/05_user_analysis.sql`

### 6.6 商品分析

商品分析主要包括：

- 最受欢迎商品：总行为量最高的商品
- 最热销商品：购买次数最高的商品
- 最有潜力商品：收藏和加购次数最高的商品
- 商品转化率：购买次数 / 浏览次数

为避免低浏览商品导致转化率失真，项目中对商品转化率设置了筛选条件：

```sql
HAVING pv_count > 100
   AND buy_count > 10
```

对应 SQL 文件：`sql/06_item_analysis.sql`

### 6.7 类目分析

类目分析主要包括：

- 最受欢迎类目
- 最热销类目
- 最有潜力类目
- 类目转化率

类目转化率同样使用浏览量和购买量门槛进行筛选，避免样本量过小造成误判。

对应 SQL 文件：`sql/07_category_analysis.sql`

### 6.8 复购分析

复购分析基于购买用户生成 `user_purchase_summary` 视图，计算：

- 购买用户总数
- 复购用户数
- 复购率

复购率计算口径为：

```text
购买 2 次及以上的用户数 / 购买用户总数
```

对应 SQL 文件：`sql/08_repeat_purchase_analysis.sql`

## 7. 核心 SQL 设计

本项目没有直接修改原始表 `userbehavior`，而是通过清洗生成新的分析表：

```sql
CREATE TABLE userbehavior_clean AS
SELECT DISTINCT
  user_id,
  item_id,
  category_id,
  behavior_type,
  time_stamp,
  FROM_UNIXTIME(time_stamp) AS event_time
FROM userbehavior
WHERE user_id IS NOT NULL
  AND item_id IS NOT NULL
  AND category_id IS NOT NULL
  AND behavior_type IS NOT NULL
  AND time_stamp IS NOT NULL
  AND behavior_type IN ('pv', 'fav', 'cart', 'buy')
  AND time_stamp BETWEEN UNIX_TIMESTAMP('2017-11-25 00:00:00')
                     AND UNIX_TIMESTAMP('2017-12-03 23:59:59');
```

同时，为提高大数据量下的查询效率，对清洗表添加了多个索引：

```sql
ALTER TABLE userbehavior_clean
  ADD INDEX idx_user_time (user_id, event_time),
  ADD INDEX idx_behavior_time (behavior_type, event_time),
  ADD INDEX idx_item_behavior (item_id, behavior_type),
  ADD INDEX idx_category_behavior (category_id, behavior_type),
  ADD INDEX idx_item_time (item_id, event_time),
  ADD INDEX idx_category_time (category_id, event_time);
```

## 8. 关键结论

根据 SQL 分析结果，可以从以下几个方向总结业务结论：

1. **用户行为以浏览为主，购买行为较低**  
   在清洗后的数据中，浏览行为（`pv`）占比最高，说明用户在平台上的主要行为仍集中在商品浏览阶段。收藏、加购和购买行为占比较低，表明从兴趣产生到最终购买之间存在明显流失。

2. **加购行为比收藏行为更接近购买转化**  
   从行为次数漏斗来看，加购行为数量高于收藏行为，且加购用户到购买用户的转化表现较好。说明加购用户具有更强的购买意向，是后续运营转化的重点人群。

3. **用户层面的购买覆盖率较高**  
   从 UV 与购买用户数来看，分析周期内有较大比例的活跃用户产生过购买行为，说明平台整体用户质量较好。但从行为次数角度看，浏览到购买的转化率仍然较低，单次浏览转化效率仍有提升空间。

4. **用户活跃时间集中在晚间**  
   按小时分析发现，用户行为高峰主要集中在 20 点至 22 点，其中 21 点和 22 点的行为次数较高。说明晚间是用户浏览、加购和购买较集中的时间段，适合安排促销活动、优惠推送和商品推荐。

5. **周末或临近活动日期用户活跃度明显提升**  
   按日期分析发现，2017-12-02 和 2017-12-03 的浏览、加购、购买等行为均明显高于前几天，说明用户活跃度存在明显时间波动，可能受到周末或活动预热影响。

6. **高活跃用户不一定是高价值用户**  
   用户活跃度 TOP20 中，部分用户行为次数较多，但主要集中在浏览，购买次数较少甚至没有购买。这说明不能只用行为总次数判断用户价值，还需要结合购买次数、加购次数、最近购买时间等指标进行综合判断。

7. **商品表现存在明显差异**  
   商品分析显示，部分商品浏览量较高但购买量不一定高，部分商品购买量较高但浏览量不一定最大。因此，商品运营不能只看流量，还应结合浏览量、加购量、购买量和转化率综合评估。

8. **类目之间的流量与购买能力差异明显**  
   类目分析显示，不同类目的浏览量、加购量和购买量存在明显差异。高浏览类目适合作为流量入口，高购买类目适合重点转化，高加购类目则适合通过优惠、提醒等方式促进成交。

9. **用户复购表现较好**  
   在产生购买行为的用户中，有较高比例用户发生了多次购买，说明平台具备一定用户粘性。后续可以针对复购用户进行会员维护、个性化推荐和优惠券触达，提高长期用户价值。
   
## 9. 运营建议

1. 针对加购未购买用户进行优惠券、降价提醒或限时促销触达，提高购买转化率。
2. 在 20 点至 22 点用户活跃高峰期安排促销活动、优惠推送和商品推荐，提高运营效率。
3. 对高浏览低购买商品进行重点优化。对浏览量较高但购买转化率较低的商品，可以从价格、商品标题、详情页、评价、运费和促销力度等方面进行优化，减少用户浏览后的流失。
4. 对高加购低购买商品进行促单转化。高加购商品说明用户已有较强购买意向，但可能仍存在价格、优惠、库存或决策犹豫等问题，可以通过购物车优惠券、限时折扣、降价提醒等方式促进成交。
5. 对热销商品增加曝光和库存保障。对购买次数较高的商品，可以增加首页推荐、搜索推荐或活动资源位，同时关注库存和供应链，避免因缺货影响成交。
6. 对受欢迎类目增加流量承接。对浏览量较高的类目，可以作为平台流量入口，增加类目页推荐、专题活动和关联商品推荐，提高用户继续浏览和转化的机会。
7. 对高潜力类目进行促销和精细化运营。对收藏、加购较多但购买转化仍有提升空间的类目，可以结合优惠券、满减活动、榜单推荐等方式刺激购买。
8. 对高转化商品和类目沉淀运营经验。对购买转化率较高的商品和类目，可以分析其价格、品牌、评价、促销方式和页面内容，将有效经验复用到相似商品和类目中。
9. 对高购买和高复购用户进行重点维护。可以通过会员权益、个性化推荐、复购优惠券等方式提升用户留存和生命周期价值。
10. 建立用户、商品、类目的分层运营体系。用户侧区分高价值用户、潜力用户和普通浏览用户；商品侧区分热销商品、受欢迎商品、潜力商品和低转化商品；类目侧区分高流量类目、高转化类目和潜力类目，从而制定更精细的运营策略。

2. **贴近真实业务场景**  
   分析内容围绕电商运营中常见的用户行为、转化漏斗、商品表现、用户分层和复购问题展开，具有较强业务应用价值。

3. **SQL 结构清晰**  
   SQL 脚本按照模块组织，每个分析问题都有明确的业务问题和分析目的，便于阅读和复现。

   项目没有直接修改原始表，而是生成清洗后的分析表，保留了原始数据的完整性。

   项目在 SQL 查询基础上进一步结合业务含义，输出了用户运营、商品运营和类目运营相关建议。

## 11. 项目不足与后续优化

目前项目仍存在以下不足：

1. 当前项目主要基于 SQL 完成分析，可视化展示较少。
2. 数据字段较少，缺少订单金额、商品价格、用户画像、店铺信息等字段，因此无法进一步分析 GMV、客单价、利润、用户生命周期价值等指标。
3. 当前分析主要属于描述性分析，暂未涉及预测模型或更复杂的用户分群方法。
4. 复购分析受分析周期限制，仅能反映短期复购情况，不能代表长期用户粘性。
5. 商品转化率和类目转化率仅基于浏览与购买行为计算，未进一步结合曝光来源、价格、活动等因素。

后续可以从以下方向优化：

1. 使用 Power BI 制作可视化看板。
2. 使用 Excel 或 Python 进行进一步的数据探索。
3. 增加留存分析、复购周期分析和用户生命周期分析。
4. 引入订单金额字段后，进一步分析 GMV、客单价和用户价值。
5. 将 SQL 查询结果整理成完整的数据分析报告，提高项目展示效果。
